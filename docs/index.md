#### デバイス登録のAPI刷新
**【課題】**
- スマホアプリへのPush通知等に使用する端末IDについて、従来はアプリ側で生成したID（identity_code）を使用していたが、サーバー側で発行・管理するID（identifier）へ移行する必要があった。既存ユーザーへの影響を最小限に抑えつつ、新方式へスムーズに切り替える設計が求められた。

**【工夫】**
- 新旧IDが混在する移行期間を考慮し、デバイス特定ロジックに優先順位を導入。
- APIレスポンスを常に新ID（identifier）とすることで、クライアントアプリ側はロジックを分岐させることなく、サーバーから返却されたIDを保存・利用するだけで新方式に対応できる疎結合な設計とした。

**【具体的な設計】**
- **APIエンドポイント新設**: `POST /api/service_name/v1/devices` を実装。
- **フォールバック探索ロジック**:
  - まず `identifier`（新ID）での検索を試行。
  - ヒットしなければ `identity_code`（旧ID）で検索。
  - いずれも存在しない場合は新規レコードを作成（`identifier`発行）。
  - このロジックにより、既存ユーザーはアクセスしたタイミングで自動的に新IDの発行・紐付けが行われる。
- **DBスキーマ変更**: `devices` テーブルに `identifier` カラムを追加。
- **管理画面（CMS）対応**: デバイス検索・詳細画面に `identifier` 表示を追加（Admin権限のみ）し、運用時の問い合わせ調査に対応。

**【成果】**
- 既存アプリユーザーの利用を阻害することなく、サーバー発行型IDへの移行基盤を確立。
- クライアントアプリの実装負荷を下げつつ、将来的な旧ID（identity_code）廃止に向けたデータ移行の道筋をつけた。

#### S3クライアントのリファクタリング
**【課題】**
- コードベース内の各所で AWS S3 クライアントのインスタンス化や操作が個別に実装されており、コードの重複や保守性の低下を招いていた。

**【工夫】**

- 共通ラッパークラス `ServiceName::Aws::S3Client` に S3 操作を集約することで、接続設定やバケット操作のロジックを一元管理できるようにした。
- これにより、S3 関連の仕様変更（例: バケット名の変更、認証情報の扱いの変更など）があった場合も1箇所の修正で対応可能とした。

**【具体的な設計】**
- **ラッパークラスの拡張**: `lib/service_name/aws/s3_client.rb` にメソッドを追加・修正し、各利用箇所で必要な機能をカバー。
- **既存コードの置き換え**:
  - `app/forms/`, `app/operators/`, `app/services/` など、散在していた S3 操作コードを特定。
  - `Aws::S3::Resource.new` 等の直接呼び出しを `ServiceName::Aws::S3Client.new` を経由したメソッド呼び出しにリファクタリング。
  - 署名付きURLの発行やファイルのアップロード処理などを共通メソッド経由に変更。
- **インターフェース統一**: アップロード、ダウンロード、署名付きURL発行などの操作を統一されたインターフェースで提供。

**【成果】**
- 40以上のファイルにわたり、合計で数百行のコード削減と重複排除を達成（削除行数が追加行数を大幅に上回っている）。
- S3 操作の責任が単一クラスに集約され、コードの可読性と保守性が向上。
- テストコード（RSpec）も含めてモック化や呼び出しの検証が容易になった。

#### アプリ利用登録フロー改善
**【課題】**
- 入居者向けアプリの利用登録において、二次元バーコードやメール招待など複数の登録手段が存在する中、管理会社側でどの手段を有効化するかを制御する必要があった。
- また、アプリ側でこれらの設定を取得し、画面表示を動的に切り替える仕組みが不足していた。

**【工夫】**
- 既存のID設定モデルと利用登録設定モデルを拡張し、各登録手段の有効/無効フラグや設定項目（メール本文、リンク先など）を追加。
- これらの設定をモバイルアプリ側へ一括提供する専用APIを新設することで、フロントエンドとバックエンドの設定同期をシンプルにした。

**【具体的な設計】**
- **DBスキーマ拡張**:
  - `id_settings` テーブルに登録書類利用フラグ (`is_registration_paper_available` 等) や自動招待メール設定のカラムを追加。
  - `registration_page_settings` テーブルに自動招待メール本文などの設定を追加。
- **API実装**:
  - 新設エンドポイント: `GET /api/service_name/v1/app_registration/settings`
  - レスポンス内容: 利用可能な登録タイプ (`available_types`: 'document', 'email')、QRコード利用可否、問い合わせ先情報などをJSONで返却。
- **管理画面改修**:
  - 企業マスタ設定画面に「利用登録書類利用」「自動招待メール利用」「お問い合わせ導線」などの設定項目を追加。
  - 入居者一覧画面において、設定がOFFの場合は関連機能（一括操作ボタンやチェックボックスなど）を非表示にする制御を追加。

**【成果】**
- 管理会社ごとの運用方針に合わせて、アプリ利用登録フロー（書面、メール、QRコード）を柔軟にカスタマイズ可能になった。
- モバイルアプリ側は単一のAPIから設定を取得するだけで、企業の契約・設定状況に応じた適切なUI/UXを提供できるようになった。
- 管理画面での設定項目追加により、運用者が自律的に登録フローをコントロールできるようになった。

#### CIの改善
**【課題】**
- RSpecの全テスト実行に約1時間を要しており、開発サイクルを鈍化させる要因となっていた。
- CodeCovのカバレッジレポート運用が機能しておらず放置状態だった。
- テストのボトルネックを特定しようにも、従来の計測ツール（深さ3, 4のみ対象）では粒度が粗く、効果的なリファクタリング対象を絞り込めなかった。

**【工夫】**
- 外部SaaS（CodeCov）への依存を排除し、CircleCIのArtifactsを活用する自律的な運用へ移行。
- さらに、実行時間計測ロジックをディレクトリ階層の深さ1〜6まで網羅的に計測・出力するように拡張し、「どこが遅いか」をピンポイントで特定できる分析基盤を構築した。

**【具体的な設計】**
- **カバレッジ計測ツールの移行**:
  - CodeCovを廃止し、SimpleCovが出力するHTMLレポートをCircleCIのArtifactsにアップロードする構成へ変更。
  - コストをかけずにチーム全員が詳細なカバレッジを確認できる環境を整備。
- **実行時間分析の強化**:
  - `lib/tasks/rspec/calculate_execution_time_percentage.rake` を改修し、深さ1〜6までのディレクトリ単位で実行時間と割合（％）を可視化。
  - この分析結果に基づき、ボトルネックとなっていたテスト（例：特定のOperatorやController spec）を特定し、集中的なリファクタリングを実施。

**【成果】**
- **テスト実行時間の大幅短縮**: ボトルネック解消により、約1時間かかっていたテスト実行時間を **20分（1/3）** にまで短縮することに成功。
- **コスト削減**: テスト時間の短縮により、CircleCIのコンピューティングリソース費用を **1回あたり約56円削減**。頻繁に実行されるCIにおいて、月次で無視できない規模のコストダウンを実現した。
- **開発効率の向上**: フィードバックサイクルの高速化により、開発チームの生産性を向上させた。

#### 契約更新確認機能
本機能は、従来の重厚な契約更新手続き（書類締結等）の前段階にある、「更新意思の確認」プロセスをデジタル化・軽量化するためのものです。

**【課題】**
- 契約満了が近づいた入居者に対し、「更新するか退去するか」の意思確認を行う業務は、電話や郵送などのアナログな手段に依存しており、管理会社の工数を圧迫していた。
- また、本格的な更新手続きフローに乗せる前に、まず意思だけを確定させたいというニーズに対し、既存システムは重すぎた。

**【工夫】**
- 「更新手続き」とは切り離された、独立かつ軽量な**「意向確認（Renewal Confirmation）」**というドメインを定義。
- 入居者はスマホアプリ/Webからワンタップで回答でき、管理会社はダッシュボードで進捗をリアルタイムに把握できる、双方にとって負荷の低い仕組みを構築した。

**【具体的な設計】**
- **デュアル・ドメインモデル**:
  - `RoomContracts`（部屋）と `ParkingContracts`（駐車場）でモデルとテーブルを物理的に分離しつつ、共通ロジックを `ContentConcern` に集約。
  - 将来的に部屋と駐車場で異なる確認フローが発生しても柔軟に対応できる疎結合な設計を採用。
- **イベントソーシング型ログ**:
  - 単なる `updated_at` 更新ではなく、`Log` 名前空間下（`StatusLog`, `AssigneeIdLog` 等）に操作種別ごとのログモデルを細かく定義。
  - 「いつ」「誰が」「何を」変更したかの完全な監査証跡を残すことで、言った言わないのトラブル防止や運用ミス追跡を可能にした。
- **通知スケジュールの予約管理**:
  - `AnnouncementReservation` モデルにより、初期通知とリマインド通知をシステムが管理。未回答者への自動リマインドを実現し、回収率向上を図る設計。
- **Elasticsearchによる検索最適化**:
  - 大量の契約データから「未回答」「期限間近」「回答済み（更新希望）」などのステータスで即座に抽出できるよう、Elasticsearchのインデックス設計を最適化。

**【成果】**
- 管理会社は、数千件規模の契約更新対象者に対する意思確認業務を自動化・効率化できる基盤を得た。
- 入居者は、煩雑な書類やり取りの前にスマホだけで意思表示が可能となり、UXが大幅に向上。
- 結果として、更新意思確認のリードタイム短縮と、管理会社の運用コスト削減を実現。

#### 汎用物件選択機能と通知最適化
**【課題】**
- 入居者アプリが未登録または未ログインのユーザーからの解約申請等において、システムが入居物件を特定できず申請通知が管理会社の全社共通窓口などに飛んでしまい、担当者の振り分け工数や対応遅れが発生していた。
- 特に、大手管理会社さんの導入において、これが業務運用上のブロッカーとなっていた。

**【工夫】**
- 解約申請フローの前に、汎用的な「物件選択」プロセスを独立したフェーズとして挿入する設計とした。
- ユーザー自身に郵便番号・住所から正規化されたシステム上の「建物」「部屋」を選択させることで、申請内容と管理会社のマスタデータを確実に紐付け、正しい担当窓口（建物別連絡先）への通知ルーティングを実現した。

**【具体的な設計】**
- **汎用物件選択フローの確立**:
  - 解約申請に限らず、利用登録や問い合わせ等にも転用可能な汎用ページ・APIとして設計。
  - ユーザー入力住所の揺らぎ（「7丁目」「7-」等）を吸収する住所正規化ロジック（`NormalizeAddressUtil`）を実装し、Elasticsearchを用いた高精度な建物検索を実現。
- **物件情報補完（PropertyInfoComplement）のデータ基盤**:
  - 選択された物件情報を一時的に保存・識別するための `PropertyInfoComplement` モデルを拡張し、API経由で `identifier` を発行する仕組みを構築。
  - この `identifier` を後続の解約申請フローに引き渡すことで、未ログインユーザーでも申請データと物件マスタの紐付けを可能にした。
- **通知ルーティングの最適化**:
  - 契約データが存在しない未登録・未ログインユーザーであっても、選択された建物に紐づく正しい通知先へメールを送信可能にした。

**【成果】**
- **業務運用の効率化**: 申請通知が適切な担当者（営業所やエリア担当）に直接届くようになり、管理会社内での転送・確認コストを削減。
- **大手案件の導入推進**: システムの導入検討中の企業の導入要件を満たし、導入障壁を解消。
- **汎用性の確保**: 物件選択機能をモジュール化（API/画面）したことで、今後、他機能への横展開も容易な基盤を構築した。

## 3. 汎用申請管理モジュールの新規構築
**【汎用申請プラットフォーム基盤開発】**
- 多様な業務（解約、問い合わせ、契約更新等）で発生する申請プロセスを一元管理する汎用モジュールの設計・開発。
- 従来の個別開発による工数肥大化を解消する「フォームビルダー機能」と、厳格な監査要求に応える「イミュータブルな履歴管理基盤」の構築を実現しました。

### チーム構成
エンジニア3名、QA1名

### 担当範囲
- プロジェクトリーダー
  - 要求仕様の策定・要件定義・技術選定・詳細設計・実装・進捗管理

### 使用技術
TypeScript, NestJS, React, Redis, PostgreSQL, Drizzle ORM, Zod, OpenAPI, Swagger, Vitest, Docker, Github Actions

**【課題】**
- 既存の申請フォームは、解約・問い合わせ・契約更新・アンケートなど、多様な申請種別に対応するため、個別の実装が必要であり、拡張性や保守性の低さ、工数の増大が課題となっていました。
  - これらの課題を解決するために、申請処理をドメインから切り離し、共通化して管理できる仕組みが求められました。
- 申請データの履歴管理については、過去の情報の整合性や監査証跡を担保しつつ、多くのシステム改修や運用コストの削減を図る必要がありました。

**【工夫】**

**1. イミュータブルデータモデルによる「事実」の保全**

「事実は書き換えられない」という原則に基づき、Updateを原則禁止する Append-Only なデータアーキテクチャを設計・実装しました。
- **IDと状態の物理分離:**
    不変の識別子を持つ「管理テーブル」と、状態スナップショットを持つ「履歴テーブル」を分離。申請内容の変更は全て新しいバージョンの `INSERT` として処理し、過去の全履歴を改変不可能な状態で保持しました。
- **完全な状態復元:**
    各バージョンにその時点の全データ（JSONB）と変更理由（Event Type）、実行者を記録。これにより、システム障害時や監査対応時に「特定日時の状態」を再現可能にしました。
- **実用性の確保:**
    Read性能への影響を最小化するため、最新状態のみを持つRead Modelを併用するCQRS的なアプローチの導入も設計に組み込んでいます。

**2. JSON Schema駆動による「開発レス」なフォーム基盤**

申請フォームの構造をJSON Schemaで定義し、それを解釈してUIとバリデーションを動的生成するエンジンを開発。
- DBスキーマ変更やAPIデプロイなしで、管理画面からの設定のみで新規フォームをリリース可能な仕組みを構築しました。

**【成果】**
- ドメイン固有の申請処理実装にかかるコスト削減を実現し、今後の運用負荷を大きく削減できる見込みです。
  - 従来必要だった個別の申請フォーム実装にかかる工数がほぼゼロになると見込まれ、人件費やリリース時間の大幅削減につながると予測しています。
- 将来的には、既存システムの申請部分の多くをこの汎用モジュールに移行する計画が確定しており、システムの保守コストと開発工数を大幅に低減し、ビジネスの迅速な拡大と柔軟な運用を支援します。
- 履歴管理の仕組みは、監査証跡やコンプライアンスにも対応可能であり、システムの信用性と信頼性を高めることに寄与しています。
- イミュータブル設計により、同時更新によるデータ競合や意図しない上書き（Lost Update）のリスクを構造的に排除しました。
- PostgreSQLのRLS（Row Level Security）を活用し、DBレベルでテナント間のデータ分離を保証するセキュアなマルチテナント構造を設計・実装しました。
